
## Ashby Autofill-widget BEATEN (chain_034, 2026-05-30 — partial SHIP)

**The chain_033 "autofill clobber" gap is SOLVED for simple tenants.** Root cause finally understood: the Ashby "Autofill from resume" widget does NOT just race-populate fields — it **REBUILDS THE FORM DOM ENTIRELY** after parsing the resume. The original `[name=<fieldId>]` inputs the chain033 runner typed into are DESTROYED and replaced with new elements (different/no stable names). That's why chain033's post-upload retype timed out on stale locators.

**The fix — upload-FIRST, fill-by-VISIBLE-LABEL** (`role-discovery/_ashby_autofill_runner_chain034.py`, prototype `/tmp/ashby_autofill_runner.py`):
1. Navigate to `/application`.
2. Upload resume to the FIRST `input[type=file]` (the autofill widget) IMMEDIATELY.
3. `wait_for_timeout(6000)` for autofill to rebuild + prefill name/email/phone from the resume.
4. Fill REMAINING fields by **visible label text** (not plan field-names, which are now stale): `findContainer(labelText)` walks DOM for an element whose normalized textContent matches the label, climbs to the field container, then fills the input / clicks the radio option / ticks the checkbox inside it.
5. Comboboxes (Ashby typeahead, `input[role=combobox]` placeholder "Start typing..."): click → `keyboard.type(query)` → wait → click `[role=option]` matching.
6. Submit + scrape for "successfully submitted".

**Usage**: `python _ashby_autofill_runner_chain034.py <inline-plan.json> '<fills_json>' [--no-submit]` where fills_json = `{"text":{"<label>":"<val>"}, "comboboxes":{"<label>":"<query>"}, "radios":{"<label>":"<option>"}, "checkboxes":{"<label>":["<opt>"]}}`.

## Queue-selection footgun: `status=''` ≠ unapplied (chain_039, 2026-05-31)
A submit chain was handed 59 "fresh Greenhouse" role-ids selected by `status=''`. **All 59 were already applied** (`applied_on` set 2026-05-08→05-26, `applied_by` auto/cyrus, real STATUS.md + confirmation URLs on disk). `inline_submit.py` correctly hard-refuses these on prep (`ValueError: role id=N already applied on <date> by <who>`), so no dup was created — but the chain wasted a dispatch.

**Root cause:** successful `auto`/`cyrus` submits do NOT always get `status` flipped to `'applied'`; many land with `status=''` + `applied_on` set. As of 2026-05-31 there were **92 rows** with `status='' AND applied_on IS NOT NULL`.

**Rules for any submit-chain dispatcher / selector:**
1. NEVER select submit candidates by `status=''` alone. ALWAYS add `AND applied_on IS NULL`. (`inline_submit.py`'s own refusal is the backstop, but it shouldn't be the only guard — it burns a chain slot per stale row.)
2. After a confirmed submit, write BOTH `applied_by`/`applied_on` AND `status='applied'`. The status flip is the field the selectors trust.
3. Periodic reconcile: `UPDATE roles SET status='applied' WHERE status='' AND applied_on IS NOT NULL;` (non-destructive; preserves applied_by/applied_on).
