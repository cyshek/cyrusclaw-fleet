## 4) VERDICT

### → **KEEP-IN-QUARANTINE**

The candidate is byte-for-byte identical to the live parent breakout_xlk in everything the harness actually backtests, at ALL three cost levels (2/5/10 bps): median_return_pct, median_sharpe, pct_positive, pct_beat_bh_spy, worst_return_pct and total_trades match the parent exactly (e.g. 2bps medSharpe +1.362 vs +1.362, 10bps +0.991 vs +0.991; total_trades 96 vs 96 at every cost). The reason is a harness/strategy interface mismatch, NOT alpha: runner/backtest.py reads action.notional_usd but never action.qty, and a 'sell' action with notional_usd=0 is rejected by the risk gate as 'non-positive notional'. The candidate's one-shot partial exit emits Action('sell', qty=holding*0.5, notional_usd=0.0), so all 20 of its scale-out attempts across the panel (3+3+5+2+2+2+2+1 by window) are REJECTED and the partial exit NEVER executes. So the candidate neither helps nor hurts -- it is a no-op clone. It cannot be PROMOTED (it would add zero edge over the already-live parent while implying a partial-exit behavior that does not exist), and it should not be REJECTED outright because the underlying idea (lock in half at the median +2.60% runup, let the rest ride) is sound and the trigger demonstrably fires on ~20 trades -- it is the EXECUTION PATH that is broken, not the hypothesis. Keep in quarantine pending a harness fix or a strategy rewrite.

**What would change the call:**

What would change the verdict to a real PROMOTE/REJECT test: (1) PREFERRED -- teach the backtester to honor Action.qty on sell/close (sell exactly qty shares, reduce cost_basis pro-rata, only pop position_state when qty hits 0, and skip the non-positive-notional gate for qty-based sells). Then re-run this exact matrix; the partial exit will fire on ~20 trades and we can measure whether the 50% de-risk actually improves median Sharpe net of the extra sell turnover (partial exits ADD fills, so 5/10bps should hurt the candidate MORE than the parent -- that is the real test this run could not perform). OR (2) rewrite the strategy to express the scale-out in a notional the harness accepts -- emit Action('sell', notional_usd=0.5*current_position_usd) instead of qty -- but note the current risk gate treats a sell's notional as a BUY-style cap check, so the strategy/harness contract for partial sells needs to be defined first. Until one of those lands, this candidate provably equals the parent and adds nothing.
