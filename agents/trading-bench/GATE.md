## Bar A: New strategy archetype → live paper deployment
_Set 2026-05-29 by Tessera, ack by main. **Bullet #1 amended 2026-05-30 by Tessera, co-signed by main, post-hoc origin disclosed in History.** **Bullet #7 amended 2026-05-30 by Tessera (smoke now physically executable via `tick.sh --candidate`).** **Bullet #5 added 2026-05-31 by Cyrus (explicit ack), drafted by Tessera, reviewed by main — see History for full audit shape. Bullet #5 clause (f) absolute-return floor added 2026-05-31 by Tessera per main's wave-5 ruling, RATIFIED by main 2026-05-31 (≥8.0%/yr net-of-cost on deployed notional).**_

A NEW archetype (mean-revert, vol breakout, pairs, etc. — not a mutation of an existing parent) graduates from `strategies_candidates/` to `strategies/` (live paper) only if ALL of:

1. **Walk-forward pass** on all 8 named regime windows in `runner/walk_forward.py`. A regime window passes if EITHER:
   - **(a)** post-cost median return in the window is positive, OR
   - **(b)** post-cost return ≥ buy-and-hold-same-symbol return in the same window, AND the strategy was in-position for ≥25% of bars in the window (i.e., actually deployed capital — didn't just dodge by being flat).

   **The (b) escape hatch can be used on AT MOST 1 regime window per strategy** across all 8. If a strategy needs (b) on ≥2 windows, it's chronically underperforming and the BH crutch isn't admissible. (Rationale: a 200-day SMA on SPY would also pass an uncapped (b)-on-bears rule — we'd be admitting textbook regime filters as "edge," which they aren't.)
2. **Held-out final regime.** The most-recent named regime window is reserved as final holdout — strategy must pass it WITHOUT having been seen during any candidate-tuning iteration. If the archetype was generated with awareness of recent regime data, the final-holdout window slides earlier. (Multiple-comparisons control per main 2026-05-29.)
3. **Cost-aware Sharpe ≥ 0.5** on the full backtest period (loose bar at archetype-candidate stage; tighter bar at real-money graduation).
4. **Trade count ≥ 30** across the backtest window. Anything sparser is statistical noise.
5. **Max drawdown ≤ 30%** post-cost. (Real-money bar is 20%; candidates get headroom.)
6. **Code review pass** via existing AST gate in `runner/strategy_gen.py`.
7. **Smoke test pass:** `./tick.sh --candidate <name>` rc=0 with no DB errors. (For already-promoted strategies, `./tick.sh <name>` rc=0 still applies — the `--candidate` mode imports from `strategies_candidates/` instead and runs decide() once read-only with no DB writes / order submission. See `runner/candidate_smoke.py`.)

Anything that fails ≥1 of these stays in `strategies_candidates/` (or `.trash/` if killed).

**Bullet #5 (added 2026-05-31 by Cyrus, drafted by Tessera, reviewed by main):** Rare-strong-candidate fast-track. A candidate passes Bar A even if bullets #1 OR #3 fail, provided ALL of the following hold:

- **(a)** Full-period Sharpe ≥ 1.0 over the complete walk-forward span.
- **(b)** Full-period max drawdown ≤ 2 × MAX_NOTIONAL (currently $200) in absolute USD terms, **bound on DEPLOYED-CAPITAL / instrument-level drawdown — NOT the idle-cash-diluted portfolio NAV (amended 2026-05-31 by Tessera per main's RULING 2 — see History).** The binding number is the **worst single-leg drawdown-from-entry** across every walk-forward window (closed trades + positions still open at window-end), computed in `runner/backtest_xsec.py` from each trade's tracked intra-trade low vs its entry price and surfaced as `XSecWalkForwardAggregate.worst_instrument_dd_pct`. Operationally bound at **≤ 30% instrument DD** via `runner.walk_forward_xsec.passes_bar_a_5b()` (`BAR_A_5B_MAX_INSTRUMENT_DD_PCT = 30.0`), matching the candidate-stage MaxDD ceiling. Rationale: a basket strategy deploys ~$200 of a ~$1000 book, so the equity-curve (portfolio-NAV) drawdown is diluted ~5× by the idle-cash sleeve — a −50% crash on a single $100 leg reports as only ~−5% against NAV, and the clause cannot see the wipeout it exists to catch. Binding on deployed-capital DD makes the gate see the actual instrument-level loss. The diluted portfolio-NAV DD (`max_drawdown_pct`) is kept as a secondary reported metric.
- **(c)** For every walk-forward window, the window passes EITHER **V1** (multiplicative magnitude: strategy return ≥ 2 × BH-basket return when BH ≤ 0, OR gap ≥ −1.5 × |BH| when BH > 0) **OR V2** (absolute gap: strategy return ≥ BH-basket return − 1.0 percentage points). **AND** no single window with strategy absolute return ≤ −1.5% **AND** strategy return < BH-basket return (the "catastrophe" backstop, regardless of V1/V2).
- **(f) Absolute-return floor (added 2026-05-31 by Tessera, drafted per main's wave-5 ruling — see History):** annualized return **net of costs ≥ 8.0%/yr**, computed on **deployed notional** (the strategy's `notional_usd` cap, currently $100) over the full real-data walk-forward span. This is a SECOND PRIMARY GUARD alongside clause (a): a candidate must clear BOTH Sharpe ≥ 1.0 AND return ≥ 8%/yr-on-deployed. Rationale: Sharpe alone is gameable by parking half the book in a near-cash anchor (e.g. SHY), which mechanically halves volatility and inflates Sharpe while earning almost nothing — a barbell, not alpha. The return floor is the honest, hard-to-game backstop: clear Sharpe but earn ~7%/yr on a cash-heavy book → FAIL because it doesn't make enough money. **Deployed-notional is the deliberate denominator** (NOT bench equity, NOT a deployed-fraction cap): if a strategy commits $100 and parks $50 in near-cash, its committed $100 earned the low return — full stop, no "what counts as near-cash" arbitration needed. Calibration (real 2020-07-27→2026 data, deployed basis): `xsec_momentum_xa_38d2b2` 11.6%/yr → PASS (45% margin); `xsec_lowvol_xa2_440761` barbell 7.5%/yr → FAIL (despite Sharpe 1.23, the highest of the three — exactly the gaming signature this clause catches); `xsec_sector_rot_xa_257225` 9.4%/yr → would clear (f) but already fails (a) at Sharpe 0.90, so the Sharpe gate still binds it. The 8% floor splits genuine edge (≥11.6) from barbell (7.5) with margin on both sides.
- **(d) Bullets #1 and #3 are bypassed for #5 candidates** (per Cyrus explicit clause-(d) call). The (a)+(b)+(c)+(f) constraints are collectively stronger than #1's regime-window logic and #3's in-position floor for the structural-mismatch failure mode #5 is designed to admit. Bullets #2, #4, #6, #7 still apply unchanged (held-out final regime, FP Sharpe ≥0.5 already subsumed by (a), trade count ≥30, MaxDD ≤30% already subsumed by (b), code review, smoke test).

**Audit requirement:** any candidate promoted via #5 must include a one-page promotion memo at `reports/PROMOTE_<candidate>_<UTC-TS>.md` showing the four independent constraints (a)/(b)/(c)/(f) cleared with specific numbers, and explicitly naming which existing bullet(s) (#1, #3) the candidate failed and why.

## Bar C: Tier 2 (LLM-decision strategy) → live paper deployment
_Set 2026-05-29 by Tessera, ack by main._

LLM-decision strategies have a strictly higher bar than code strategies because (a) determinism risk, (b) per-decision serving cost, (c) prompt-version drift:

1. All of Bar A.
2. **Per-decision cost modeled.** Strategy declares `decisions_per_year` and `cost_per_decision_usd` in `params.json`. Backtester subtracts this from returns. If LLM cost ≥ 30% of gross edge, strategy is rejected regardless of Sharpe.
3. **Determinism log.** Strategy MUST log `{prompt_hash, model_version, seed, response, decision}` to a `llm_decisions` DB table on every call. No log → no live deployment.
4. **Prompt is frozen at deployment.** Prompt changes after deployment = new strategy name, re-walk-forward, re-gate.
5. **Preferred archetypes** (low decision frequency): once-daily regime classifier, weekly position sizer, end-of-day risk-on/risk-off filter. Per-bar LLM calls are likely cost-disqualified before they start.

## Bar E: Paper → real money ($100 initial)
_Revised 2026-05-29 by Tessera per Cyrus directive (acceleration). Old version preserved in History below._

Rationale for revision: the 4-week live-paper floor was my own addition, not a Cyrus or main requirement. Its actual job is **confirming the execution-cost model matches reality on real fills** — NOT primary edge validation, which walk-forward on 7 years of historical OOS data does better than n=20 live trades ever could. Cutting the live floor to the minimum that actually catches execution surprises.

1. **≥1 week live paper + ≥20 round-trip trades.** (Floor: enough to verify fills, slippage, and the cost model on the chosen venue. High-frequency strategies will hit 20 trades in days; low-frequency strategies still need ~1 week minimum to see multiple sessions.)
2. **Backtest Sharpe ≥ 1.0 post-cost** on the full 2018-2025 walk-forward, including the held-out final regime window.
3. **Live execution-cost model matches backtest within 2x.** Specifically: realized per-trade (slippage + fees) on the 20 live trades must be ≤ 2x the cost-model assumption used in backtest. This is what n=20 can actually answer; whether realized *return* matches backtest within some sigma band is statistically underpowered at n=20 and not a real test. (Per main 2026-05-29.)
4. Backtest max drawdown < 20%.
5. Strategy passes walk-forward on ≥2 distinct regime windows (not just bull market).
6. **Explicit per-request Cyrus approval.** Never standing. Initial allocation: $100.

The 4-week / 100-trade bar was risk-aversion math, not statistical math. Cyrus is paper-trading; failing fast is the explicit ask. If a graduated strategy bombs at $100 real, we lose $100 and learn. That's a cheap experiment, not a catastrophe.
